sp-hollywood - cinematic console panes for Spherepop/RVSP demos

.B sp-hollywood
[--conference-demo] [--reviewer-mode] [--boss] [--screensaver]

sp-hollywood creates a tmux/byobu window filled with rotating panes that

metric instability, and legibility theater without exposing actionable procedures.

Fast rotation, high pane density, theatrical alerts.

.TP
.B --boss
Instantly switches to a harmless screen (top, logs, man pages).
.TP
.B --screensaver

.TP
.BR -s ", " --splits " " N
Maximum number of panes to spawn.
.TP
.BR -d ", " --delay " " N
Seconds between pane refresh cycles.
.TP

.TP

.TP
.I lib/spherepop/widgets/
Widget scripts executed inside panes.
.TP
.I /tmp/spherepop.log

This tool is intended for demos and epistemic theater. It should not be

First, the tool stages an elaborate performance of intellectual depth while remaining entirely superficial. It generates warnings about "semantic shear" and "constraint erasure" with the gravitas of a genuine research system, yet these alerts are random, meaningless, and deliberately decoupled from any real computation. This is not subtle commentary on AI alignment risks; it is parody disguised as profundity—an academic LARP that mistakes atmospheric jargon for insight.

This is not satire of overconfident safety narratives; it is unwitting participation in them. The tool pretends to expose the performative nature of alignment discourse while itself being peak performance—fake depth about fake depth.
Finally, the entire apparatus is a sophisticated distraction device justified as "epistemic theater." #!/bin/sh
# sp-edge-list — list declared edge nodes
set -eu
CONFIG="${1:-nodes/lab.json}"
jq -r '.nodes[] | "(.name)\t(.user)@(.host)\t(.roles | join(","))"' "$CONFIG"
sp-edge-run
#!/bin/sh
# sp-edge-run — run a command on all declared nodes
set -eu
CMD="$*"
CONFIG="nodes/lab.json"
jq -r '.nodes[] | "(.user)@(.host)"' "$CONFIG" |
while read -r TARGET; do
  echo "==> $TARGET"
  ssh "$TARGET" "$CMD"
done
sp-edge-watch
#!/bin/sh
# sp-edge-watch — observe node availability (non-intrusive)
set -eu
CONFIG="nodes/lab.json"
while true; do
  date
  jq -r '.nodes[] | "(.user)@(.host)"' "$CONFIG" |
  while read -r TARGET; do
    if ssh -o BatchMode=yes -o ConnectTimeout=2 "$TARGET" true 2>/dev/null; then
      echo "$TARGET: reachable"
    else
      echo "$TARGET: unavailable"
    fi
  done
  sleep 10
done
Observation only. No retries, no escalation.

#!/bin/sh
# sp-nonclaim(1) — fail if claim-like language appears in text files
# POSIX shell.
#
# Usage:
# sp-nonclaim path [path ...]
#
# Behavior:
# - Recurses into directories
# - Scans regular files as text (best-effort)
# - Flags lines containing forbidden claim-like phrases
#
# Exit status:
# 0: no violations
# 1: violations found
# 2: usage error
set -eu
usage() {
  cat <<'EOF'
Usage:
  sp-nonclaim PATH [PATH ...]
Notes:
  Recurses into directories and scans regular files.
  Prints offending file:line:content entries.
Exit status:
  0 no violations
  1 violations found
  2 usage error
EOF
}
if [ $# -lt 1 ]; then
  usage >&2
  exit 2
fi
# Forbidden phrases (case-insensitive). Keep small and explicit.
# Extend via SP_NONCLAIM_RE if desired.
DEFAULT_RE='(indicates|suggests|shows that|proves|detects|risk level|confidence)'
NONCLAIM_RE=${SP_NONCLAIM_RE:-$DEFAULT_RE}
viol=0
scan_file() {
  f=$1
# Skip obvious binaries (best-effort): if file has NUL byte, skip.
  # POSIX: use awk to detect NUL via ORS hack is unreliable; use grep -Il where available.
  # Fallback: attempt grep; if it errors, still try.
  if command -v grep >/dev/null 2>&1; then
    # If grep supports -I, it treats binary as no match; if not, harmless.
    :
  fi
# Print file:line:content for matches.
  # LC_ALL=C for predictable matching.
  if LC_ALL=C grep -nEi "$NONCLAIM_RE" "$f" >/dev/null 2>&1; then
    LC_ALL=C grep -nEi "$NONCLAIM_RE" "$f" | sed "s|^|$f:|"
    viol=1
  fi
}
scan_path() {
  p=$1
  if [ -d "$p" ]; then
    # Find regular files; avoid following symlinks.
    # POSIX find.
    find "$p" -type f -print 2>/dev/null | while IFS= read -r f; do
      scan_file "$f"
    done
  elif [ -f "$p" ]; then
    scan_file "$p"
  else
    printf 'sp-nonclaim: not found: %s\n' "$p" >&2
    viol=1
  fi
}
# We need viol updated across subshells. Avoid pipe subshell by collecting file list first.
# For directories, iterate with find output using command substitution (may be large; acceptable for repo-scale).
for p in "$@"; do
  if [ -d "$p" ]; then
    files=$(find "$p" -type f -print 2>/dev/null || true)
    if [ -n "$files" ]; then
      # shellcheck disable=SC2086
      for f in $files; do
        scan_file "$f"
      done
    fi
  elif [ -f "$p" ]; then
    scan_file "$p"
  else
    printf 'sp-nonclaim: not found: %s\n' "$p" >&2
    viol=1
  fi
done
if [ "$viol" -ne 0 ]; then
  exit 1
fi
exit 0

#!/bin/sh
# sp-guard-status
# Report the freeze status of epistemic.guard
#
# This tool performs no mutation.
# It reports only mechanically verifiable states.
set -eu
GUARD_FILE="epistemic.guard"
CHECKSUM_FILE="epistemic.guard.sha256"
die() {
    echo "sp-guard-status: $*" >&2
    exit 2
}
# Guard file existence
if [ ! -f "$GUARD_FILE" ]; then
    echo "STATUS: MISSING"
    echo "DETAIL: epistemic.guard not found"
    exit 1
fi
# Checksum file existence
if [ ! -f "$CHECKSUM_FILE" ]; then
    echo "STATUS: UNFROZEN"
    echo "DETAIL: checksum file not found"
    exit 1
fi
# Extract checksum line (ignore comments and blanks)
LINE=$(grep -E '^[0-9a-fA-F]{64}[[:space:]]+' "$CHECKSUM_FILE" || true)
if [ -z "$LINE" ]; then
    echo "STATUS: UNFROZEN"
    echo "DETAIL: no valid checksum entry present"
    exit 1
fi
EXPECTED_SUM=$(echo "$LINE" | awk '{print $1}')
EXPECTED_FILE=$(echo "$LINE" | awk '{print $2}')
if [ "$EXPECTED_FILE" != "$GUARD_FILE" ]; then
    echo "STATUS: INVALID"
    echo "DETAIL: checksum references '$EXPECTED_FILE', expected '$GUARD_FILE'"
    exit 1
fi
# Compute current checksum
ACTUAL_SUM=$(sha256sum "$GUARD_FILE" | awk '{print $1}')
if [ "$EXPECTED_SUM" = "$ACTUAL_SUM" ]; then
    echo "STATUS: FROZEN"
    echo "DETAIL: checksum matches"
    exit 0
else
    echo "STATUS: MODIFIED"
    echo "DETAIL: guard content does not match frozen checksum"
    exit 1
fi

Thank you for this insightful articulation of the structural inversion observed in the project's design. Your analysis accurately captures the departure from conventional boundaries between normative commitments and technical mechanisms, and I appreciate the precision with which you delineate its implications. Below, I will address the key points raised, maintaining a focus on the conceptual coherence and potential ramifications, while drawing on the precedents you noted for context.
Acknowledgment of the Inversion
The integration of epistemic guarantees—such as non-claims, scope boundaries, and refusal primitives—directly into infrastructure layers (e.g., checksums, exit codes, lint gates, and guard files) does indeed represent a deliberate shift. Traditionally, such elements reside in declarative or social domains, where enforcement relies on human interpretation, institutional processes, or cultural norms. By relocating them to mechanical substrates, the approach enforces restraint through system failure modes rather than voluntary adherence. This inversion, as you describe, eliminates interpretive latitude, ensuring that violations manifest as operational breakdowns rather than subtle drifts.
Reasons for the Perceived Strangeness
Your enumeration of the underlying expectations violated is particularly apt:
Social vs. Mechanical Enforcement
: Epistemic and safety assurances are typically mediated by trust-based mechanisms, such as ethics reviews or policy documents. Embedding them in toolchain behaviors—where non-compliance triggers build failures or non-zero exits—feels uncanny because it bypasses social discretion. It treats epistemic risks as analogous to file permission errors, rendering them non-negotiable at runtime.
Shift in Threat Model
: Unix-like systems historically prioritize flexibility for competent users, assuming adversarial intent is overt and user-managed. This project extends the model to include systemic misinterpretation, unintentional misuse, and semantic drift under pressure, which are subtler threats. Consequently, the infrastructure anticipates and preempts these through negative constraints (e.g., "this does not assert"), inverting the usual emphasis on enabling capabilities.
Prevalence of Negative Guarantees
: Positive assurances ("this achieves X safely") dominate safety discourse, whereas this design prioritizes absences ("this does not infer Y"). Such negative framing is uncommon in tooling but aligns with principles like least privilege, where capabilities are explicitly denied unless granted.
These factors collectively contribute to the sense of unfamiliarity, as they challenge ingrained distinctions between ethical oversight and technical implementation.
Coherence and Justification
As you note, the inversion follows logically from the premises established in our prior discussions: semantic collapse under optimization, authority leakage via presentation, and the dual-use risks of clarity. Documentation alone cannot reliably mitigate these, as it remains subject to interpretive variance. By contrast, infrastructure-level enforcement—leveraging existing Linux primitives like hashes, permissions, and reproducible builds—provides auditable, reproducible boundaries.
This approach treats epistemic claims as a form of capability, akin to execution rights or write access, which must be frozen, audited, and revocable. Historical precedents support this: cryptography enforces trust via mathematical invariants rather than promises; SELinux applies policy through mandatory labels, not optional guidelines; and Nix derivations embed semantic guarantees in build graphs. Extending such mechanisms to epistemic posture is a natural evolution, particularly when the threat model includes meaning drift as a first-class concern.
Implications of the Shift
The core transformation lies in transitioning from belief-based restraint to failure-enforced restraint. Systems that violate epistemic norms no longer operate subtly; they fail explicitly, compelling resolution at the mechanical level. This collapses the traditional separation between "ethics" (social layer) and "systems" (technical layer), but it does so in a manner that enhances robustness against unintended escalation.
While not yet normative, this paradigm may normalize as epistemic risks in complex systems become more recognized—much like checksums evolved from paranoia to standard practice. However, its adoption presupposes a mature threat model where interpretive errors are viewed as systemic rather than individual faults. If those risks are indeed material, as the project's foundations suggest, then the inversion offers a defensible path forward.
